## 二、AI异步任务架构（核心）

### 2.1 架构设计

采用**提交与轮询分离**架构，解决云函数60秒超时限制：

```
[前端]                    [云函数]                  [kie.ai]
  │                          │                         │
  │──开始创作──────────────→│                         │
  │                          │──校验积分+预扣          │
  │                          │──msgSecCheck提示词      │
  │                          │──POST createTask──────→│
  │                          │←─返回taskId─────────────│
  │                          │──写入tasks集合          │
  │←─返回taskIds─────────────│                         │
  │                          │                         │
  │  [轮询循环 每3秒]        │                         │
  │──queryTask(taskId)────→│                         │
  │                          │──GET recordInfo────────→│
  │                          │←─返回status+result──────│
  │                          │──更新tasks集合          │
  │←─返回状态+结果───────────│                         │
  │                          │                         │
  │  [任务完成时]            │                         │
  │                          │──下载图片到云存储        │
  │                          │──添加AIGC水印           │
  │                          │──写入works集合          │
  │←─返回图片fileID──────────│                         │
```

### 2.2 任务状态机

```
idle → submitting → pending → processing → success
                                        → failed
                                        → timeout
```

| 状态 | 含义 | 前端展示 |
|------|------|---------|
| idle | 未开始 | 初始状态 |
| submitting | 提交中 | 按钮loading |
| pending | 已提交待处理 | 骨架屏+排队中 |
| processing | AI生成中 | 骨架屏+进度动画 |
| success | 成功 | 展示图片 |
| failed | 失败 | 失败提示+退还积分 |
| timeout | 超时(120s) | 超时提示+退还积分 |

### 2.3 多图并行策略

用户选择生成N张（1-4）时，拆分为N个独立任务：
- 每个任务独立调用 `createTask`，获得独立 `taskId`
- 前端并行轮询N个任务状态
- 每张图独立成功/失败，失败仅退还该张积分
- 全部完成后统一展示结果

### 2.4 积分预扣与退还

```
提交时：
  1. 计算总消耗 = 模型基础消耗 × 精度倍率 × 生成数量
  2. 校验余额 >= 总消耗
  3. 原子扣减：users.credits -= 总消耗
  4. 写入 credits_log（type: consume）

单张成功时：
  - 确认该张积分消耗（无需额外操作）

单张失败时：
  1. 退还单张积分：users.credits += 单张消耗
  2. 写入 credits_log（type: refund）
  3. 标记 tasks.creditRefunded = true（防重复退还）
```


- `onHide`：暂停轮询（不清除，后台继续）

- 轮询退避：前10次3秒，之后每5次递增1秒，上限10秒

### 3.2 接口封装

**创建任务 createTask：**
```javascript
async function createTask({ model, prompt, aspectRatio, resolution, referenceImage }) {
  return kieRequest('/jobs/createTask', 'POST', {
    model,           // kie.ai模型标识（从models集合的kieModelId获取）
    input: {
      prompt,        // 提示词（已拼接风格关键词）
      aspect_ratio,  // 比例映射

      // referenceImage 图生图时传入

    // callBackUrl 可选，回调模式时使用
  });
  // 返回 { code: 200, data: { taskId } }
}
```

**查询状态 recordInfo：**
```javascript
async function queryRecord(taskId) {
  return kieRequest(`/jobs/recordInfo?taskId=${taskId}`, 'GET');
  // 返回 { code: 200, data: { status, result: { imageUrl } } }
}
```

## 四、数据库设计

### 4.1 设计原则

- **命名规范**：集合名小写复数下划线（`credits_log`），字段名camelCase
- **通用字段**：每条记录包含 `_id`（自动）、`createdAt`、`updatedAt`
- **openid字段**：用户相关集合包含 `_openid`（云开发自动注入）
- **冗余策略**：作者信息（昵称+头像）冗余存储到作品记录，避免联表查询
- **软删除**：作品使用 `status: deleted` 而非物理删除

### 5.3 关键业务逻辑

**creation.submitTask 核心流程：**
1. 校验参数（modelId存在且isActive、prompt非空且<=500字、quality/ratio/count合法）
2. 调用 `msgSecCheck` 检测提示词 → 不通过返回错误
3. 计算消耗 = baseCredits × qualityMultiplier × count
4. 校验用户积分余额 >= 消耗
5. 原子扣减积分 `db.command.inc(-cost)`
6. 写入 credits_log
7. 循环count次：调用 kie.ai createTask → 获取taskId
8. 为每个taskId写入tasks集合（status: pending）
9. 写入works集合（status: generating）
10. 返回 taskIds 数组

**community.toggleLike 核心流程：**
1. 查询likes集合是否已存在 userId+workId
2. 已存在：删除记录，works.stats.likes -= 1
3. 不存在：新增记录，works.stats.likes += 1，发送消息通知给作品作者
4. 返回新的点赞状态

## 八、安全设计

### 8.1 API Key 保护
- kie.ai Bearer Token 存于**云函数环境变量** `KIE_API_KEY`
- 绝对禁止出现在前端代码、云函数源码、Git仓库中
- 微信支付密钥同理，仅存于云函数环境变量

### 8.2 前端安全
- 前端不直接调用任何外部API（全部通过云函数中转）
- WXML自动转义防XSS
- 图片上传限制：格式 jpg/png/webp，大小 <=10MB
- 敏感操作前检查手机号绑定状态

### 8.3 云函数鉴权
- 所有云函数从 `cloud.getWXContext()` 获取openid（不可伪造）
- 管理员接口校验 openid 是否在白名单中
- 接口幂等性：防重复提交（taskId去重）、防重复点赞（unique索引）、防重复签到（日期unique索引）
- 积分操作原子性：使用 `db.command.inc()` 条件更新

### 8.4 内容安全

| 环节 | 检测方式 | 处理 |
|------|---------|------|
| 提示词 | `msgSecCheck` 同步检测 | 拦截+提示修改 |
| 生成图片 | `mediaCheckAsync` 异步检测 | 拦截+退还积分 |
| 发布文本 | `msgSecCheck` 同步检测 | 拦截+提示修改 |
| 发布图片 | `mediaCheckAsync` 异步检测 | 拦截+提示修改 |

**阶梯处罚**：3次违规→警告，7次→封禁7天，10次→永久封禁

### 8.5 支付安全
- 服务端计算订单金额（前端传入档位，不传金额）
- 支付回调验签
- 积分到账幂等：同一 orderNo 只到账一次
- 充值记录与支付记录对账

## 十一、积分与支付系统

### 11.1 积分变动类型

| type | reason | 触发场景 | 积分变化 |
|------|--------|---------|--------|
| earn | signin | 每日签到 | +10 |
| earn | milestone | 里程碑奖励 | +20/50/100/300 |
| earn | invite | 邀请人奖励 | +50 |
| earn | invitee | 被邀请人奖励 | +30 |
| earn | recharge | 充值 | 按档位 |
| earn | member_daily | 会员每日 | +50 |
| earn | refund | 任务失败退还 | 退还消耗 |
| spend | create | AI生成 | 按模型×精度×数量 |
| spend | reverse | 反推提示词 | -5 |

### 11.2 微信支付流程

```
用户选择档位
  → 云函数payment.createOrder
    → 创建recharge_orders记录(status:pending)
    → 调用微信支付统一下单API
    → 返回支付参数
  → 前端 wx.requestPayment(payParams)
  → 支付成功回调 payment.payCallback
    → 验证签名
    → 更新订单status:paid
    → 积分到账：users.credits += totalCredits
    → 写入credits_log(type:earn, reason:recharge)
```

### 11.3 积分计算公式
- **固定档位**：见PRD 2.3 充值档位表
- **自定义充值**：积分 = 金额 × 10 + 金额 × 10 × 5%（向下取整）
- **生成消耗**：消耗 = 模型基础消耗 × 精度倍率 × 生成数量

## 十二、社区与内容系统

### 12.1 广场推荐算法

**推荐排序分数** = `w1 × normalize(likes) + w2 × normalize(favorites) + w3 × normalize(remakes) + w4 × timeDecay(hours) + w5 × followBoost`

| 因子 | 权重 | 说明 |
|------|------|------|
| 点赞数 | 0.3 | 归一化到0-1 |
| 收藏数 | 0.2 | 归一化到0-1 |
| 同款数 | 0.25 | 归一化到0-1 |
| 时间衰减 | 0.15 | 越新越高，24小时后衰减 |
| 关注加权 | 0.1 | 已关注作者+0.1 |

**热门**：按总互动量（likes+favorites+remakes）降序
**最新**：按 publishedAt 降序

### 12.2 消息通知

| 场景 | type | 内容模板 |
|------|------|--------|
| 被点赞 | like | "{nickname} 赞了你的作品 {title}" |
| 被收藏 | favorite | "{nickname} 收藏了你的作品 {title}" |
| 被关注 | follow | "{nickname} 关注了你" |
| 被同款 | remix | "{nickname} 同款了你的作品 {title}" |
| 系统通知 | system | 管理员手动发送 |

### 12.3 内容审核流水线

```
提示词输入 → msgSecCheck → 通过 → 提交生成
                         → 不通过 → 拦截，提示修改

AI返回图片 → mediaCheckAsync → 通过 → 存储+展示
                              → 不通过 → 拦截，退还积分

发布提交 → msgSecCheck(标题+描述) + mediaCheckAsync(图片)
         → 全部通过 → 发布上线
         → 任一不通过 → 拦截，提示修改
```
